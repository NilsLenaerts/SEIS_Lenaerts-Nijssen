Paper analysis

Last edit: May 30, 2022

Introduction

This blog post gives an overview of the paper Leveraging webassembly for numerical javascript code virtualization. This post also includes a short review.

Overview

This paper goes in on how webassembly(WASM) can improve the security and make reverse engineering of code harder for web-based applications. This is to prevent unauthorized use of software, cheating and the bypass of security features. Currently the most popular choice is to utilize JavaScript (JS), This comes with two challenges. It is an interpreted languages and the code is often shipped as plain source code. This high level code provides much information that can be utilized for reconstruction even with the use of code obfuscation. This also limits the strength of protection that JavaScript can provide. Because JavaScript is interpreted and just in time (JIT) compiled it also comes with larger overhead. Webassembly however is ahead of time (AOT) compiled allowing near native speed. The lower level code that is shipped is also harder to analyze, it also can be optimized pre shipment resulting in less overhead. The translation from JS to WASM however is difficult, one reason is that webassembly is strongly typed and the type must be known at compile time while JS is weakly and dynamically typed. JSPro uses webassembly to protect Javascript code, this is done by dividing the program into computational code which is compiled to webassembly and DOM code which will interact with the webassembly API. This approach lead to a 15-38% faster runtime, and a 28% reduction in code size. Which is significant as it also comes with better security. The tested code it runned on a virtual machine that can handle the split javascript code. This machine contains four components. The dispatcher, VMData, Handler and VMContext.

VMContext: a classic binary virtual interpreter. this makes the stack and registers able to participate in computation.

VMData: the custom bytecode generated based on the instructions. These represent execution logic of the original program.

Handler: Is used to restore the VMData.

Dispatcher: invokes the handlers to restore the program functions based on VMData.

Webassembly is however not designed to support object types, this serves as a limitation for javascript as this can contain various object types. In this case performance improvement is not to be expected.

Review

Personally i believe that the proposed solution can be influencial in the future of webassembly and web-based applications. Mainly for it's security features as this has become more important in the last couple of year. The improvement in at runtime speed is also a benefit that will get attention as this is also something to consider when it comes to user friendlyness of applications (personally i'm not a fan of long waiting times on web pages and this can improve that).
